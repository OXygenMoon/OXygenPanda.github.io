IO多路复用

学习自 : https://www.bilibili.com/video/BV1qJ411w7du?from=search&seid=6979152329662416408

前言

服务器应对并发时, 如果使用多线程程序, 每一个连接创建一个线程, 由于上下文切换, 处理句柄会导致代价很高. 所以很多时候, 使用的是单线程程序, 其中 DMA 保证了不同时候到来的 IO数据 不会丢失.

最简单的一个由程序来接收IO数据的程序.

while(1) {
	for(Fdx in (FdA ~ FdE) {
		if(Fdx 有数据) {
			读Fdx;
			处理;
		}
	}
}

但是, 每一次由用户来询问内核, 开销较大, 因此有了 select().

select()

while(1) {
	FD_ZERO(&rset);
	for(i = 0; i < 5; i ++) {
		FD_SET(fds[i], &rset);
	}

	puts("round again");
	select(max + 1, &rset, NULL, NULL, NULL); //阻塞函数
	/*
	para 0:文件描述符最大的范围(因为第一位是fd=0,所以要加1)
	para 1:读文件描述符(比较关注)
	para 2:写文件描述符
	para 3:异常文件描述符
	para 4:超时时间
	*/
	for(i = 0; i < 5; i ++) {
		if(FD_ISSET(fds[i], &rset)) {
			memset(buffer, 0, MAXBUF);
			read(fds[i], buffer, MAXBUF);
			puts(buffer);
		}
	}
}

工作原理

  1. 使用 rset(bitmap) 来标记注册的 Fds ;
  2. 将 rset 拷贝至内核态 ;
  3. select() 阻塞 ;
  4. 有数据到来后, 内核对有数据的 rset位 置位, 并且 select() 返回 ;
  5. 遍历 Fds 判断是否被置位, 如果置位就说明对应的 IO 有数据 ;
  6. 由于破坏了 rset位图, 需要重新遍历 Fds 标记 rest位图 ;
  7. 循环 2 - 6.

注意事项

  1. rset 的数据结构以及存储的数据的表示
rset的数据类型是 bitmap

假设 fds 分别是 1, 2, 5, 7, 9
那么rest 的值就是 bitmap<1024>0110'0101'01000...

//左边第一个代表的是 fds 值为0的不存在
//1024 因为 Linux 默认支持最大线程数是1024
  1. 工作状态
无数据 : select() 阻塞
有数据 : 内核 1. 对有数据来的FD置位 2. select() 返回
  1. 返回值
成功 : 大于 0 
出错 : -1
超时 : 0

select() 的优点

  1. 将 rset 交给内核监听, 节约了一定的开销;
  2. select() 的核心是内核等待队列唤醒机制.

select() 的缺点

  1. rset : 1024 bitmap
  2. FDset 不可重用
  3. 用户态拷贝的时候切换到内核态, 仍然有开销
  4. O(n) 时间复杂度去遍历得到哪一个 IO 有数据

poll()

struct pollfd {
	int id;        //fds
	short events;  //关注的事件
	short revents; //反馈(初始值为0)
};

for(i = 0; i < 5; i++) {
	memset(&client, 0, sizeof(client));
	addrlen = sizeof(client);
	pollfds[i].fd = accept(sockfd, (struct sockaddr*)&client, &addrlen);
	pollfds[i].events = POLLIN;
}
sleep(1);

while(1) {
	puts("round again");
	poll(pollfds, 5, 50000);   //阻塞
	/*
	para 0 : 结构体数组名
	para 1 : fds的个数
	para 2 : 超时时间
	*/

	for(i = 0; i < 5; i ++) {
		if(pollfds[i].revents & POLLIN) { //判 revents 和 关注的事件
			pollfds[i].revents = 0;         //置位达到可重用效果
			memset(buffer, 0, MAXBUF);
			read(pollfds[i], fd, buffer, MAXBUF);
			puts(buffer);
		}
	}
}

工作原理

用户态拷贝 fds 至内核态, 由内核监听(和 select() 一样)

注意事项

  1. struct pollfd 的细节
struct  pollfd 的一个节点相当于是 rset 中的一位
events  用于标记关注的事件, 如果同时关注多个事件, 事件1 & 事件2
revents 初始为0, 内核置数, 处理时清零, 可重用
  1. 工作状态
无数据 : poll() 阻塞
有数据 : 内核 1. pollfds.revents 置位 2. poll 返回

poll() 的优点

  1. 数组可以超过 1024 长(最大65535长)
  2. 重置 revents 后可重用

epoll()

struct epoll_event events[5];
int epfd = epoll_create(10);                        // 1
...
...
for(i = 0; i < 5; i ++) {
	static struct epoll_event ev;
	memset(&client, 0, sizeof(client));
	addrlen = sizeof(client);
	ev.data.fd = accept(sockfd, (struct sockaddr*)&client, &addrlen);
	ev.events = EPOLLIN;
	epoll_ctl(epfd, EPOLL_CTL_ADD, ev.data.fd, &ev);  // 2
}

while(1) {
	puts("round again");
	nfds = epoll_waits(epfd, events, 5, 10000);       // 3
	
	for(i = 0; i < nfds; i ++) {
		memset(buffer, 0, MAXBUF);
		read(events[i].data.fd, buffer, MAXBUF);
		puts(buffer);
	}
}

工作流程

1 : 创建白板 epfd

2 : 在白板上写字 : fd - events, 得到了一些列 fd - events 的 epfd

3 : epfd 由用户态和内核态共享内存 ( 实际上还是要拷贝到内核态, 在 epoll_ctl() 就拷贝 )

注意事项

  1. 函数细节
epoll_ctl()  向内核注册新的描述符或者改变某个文件描述符的状态, 已注册的描述符在内核中被维护成红黑树
epoll_wait() 回调函数, 内核将有数据的描述符加入到一个链表中, 进程获得描述符数
  1. 工作状态
没数据 : epoll() 阻塞(水平触发下, 边缘触发?)
有数据 : 1. "置位"(重排,将有数据的fd-events放在最前面) 2. 返回值是有数据的IO数
  1. 两种触发模式
LT:水平触发
	当 epoll_wait() 检测到描述符事件到达时，将此事件通知进程，进程可以不立即处理该事件，下次调用 epoll_wait() 会再次通知进程。是默认的一种模式，并且同时支持 Blocking 和 No-Blocking

ET:边缘触发
	和 LT 模式不同的是，通知之后进程必须立即处理事件。下次再调用 epoll_wait() 时不会再得到事件到达的通知。很大程度上减少了 epoll 事件被重复触发的次数，因此效率要比 LT 模式高。只支持 No-Blocking，以避免由于一个文件句柄的阻塞读/阻塞写操作把处理多个文件描述符的任务饿死。

epoll() 的优点

  1. 时间复杂度 O(1)
